# AToken

`AToken` is the interest-bearing ERC20 token used by Aave V1 to represent a user's deposit in a reserve.

When a user deposits an asset:

```text
user deposits the underlying asset
        ↓
LendingPoolCore holds the asset
        ↓
AToken mints an equivalent amount of aTokens
```

For example:

```text
deposit 100 DAI
receive 100 aDAI
```

The aToken balance increases over time as the reserve earns interest.

A useful mental model is:

```text
LendingPool
    = coordinates the deposit

LendingPoolCore
    = stores the reserve state and normalized income

AToken
    = represents the user's deposit and accrued interest
```

# Main Responsibilities

The current implementation of `AToken` has five main responsibilities:

```text
1. Mint aTokens when a user deposits.
2. Track each user's reserve-income checkpoint.
3. Calculate the user's current interest-bearing balance.
4. Materialize accrued interest during state-changing actions.
5. Track redirected interest balances.
```

# Protocol Relationships

## LendingPool

`LendingPool` is the only contract allowed to call privileged aToken functions such as:

```text
mintOnDeposit()
```

This prevents users from minting aTokens without depositing the underlying asset.

## LendingPoolCore

`LendingPoolCore` stores the reserve liquidity index and exposes:

```solidity
getReserveNormalizedIncome()
```

`AToken` uses this value to calculate how much the user's deposit has grown.

## Underlying Asset

Each aToken represents one underlying reserve asset.

Examples:

```text
aDAI represents DAI
aUSDC represents USDC
```

The underlying asset itself is held by `LendingPoolCore`, not by `AToken`.

# Access Control

## `onlyLendingPool`

```solidity
modifier onlyLendingPool() {
    if (msg.sender != address(i_pool)) {
        revert AToken__OnlyLendingPool();
    }
    _;
}
```

Only the configured `LendingPool` can call protected functions.

The current implementation uses this modifier on:

```text
mintOnDeposit()
```

# Stored State

## `i_underlyingAssetAddress`

```solidity
address private immutable i_underlyingAssetAddress;
```

Stores the underlying asset represented by the aToken.

For example:

```text
aDAI -> DAI address
```

The address is used when requesting the reserve normalized income from `LendingPoolCore`.

## `i_underlyingAssetDecimals`

```solidity
uint8 private immutable i_underlyingAssetDecimals;
```

Stores the decimals of the underlying asset.

The aToken overrides `decimals()` so that it uses the same decimal precision as the represented asset.

## `s_userIndexes`

```solidity
mapping(address user => uint256 lastNormalizedIncome)
    private s_userIndexes;
```

Stores the reserve normalized income last applied to each user.

The user index acts as a checkpoint.

For example:

```text
user index = 1.02 ray
current normalized income = 1.08 ray
```

Only the growth between `1.02` and `1.08` still needs to be applied to that user.

## `s_interestRedirectionAddresses`

```solidity
mapping(address => address)
    private s_interestRedirectionAddresses;
```

Stores the address receiving the interest generated by a user's balance.

A zero address means that the user is not redirecting interest.

## `s_redirectedBalances`

```solidity
mapping(address => uint256)
    private s_redirectedBalances;
```

Stores the amount of principal redirected to a user for interest calculation.

The recipient does not own this principal. They only receive the interest generated by it.

## `i_addressesProvider`

```solidity
LendingPoolAddressesProvider
    private immutable i_addressesProvider;
```

Stores the protocol addresses provider.

It is used during construction to retrieve `LendingPoolCore` and `LendingPool`.

## `i_core`

```solidity
LendingPoolCore private immutable i_core;
```

Stores the `LendingPoolCore` contract.

It is used to read the current reserve normalized income.

## `i_pool`

```solidity
LendingPool private immutable i_pool;
```

Stores the `LendingPool` contract.

It is used by the `onlyLendingPool` modifier.

# Constructor

```solidity
constructor(
    address _addressesProvider,
    address _underlyingAsset,
    uint8 _underlyingAssetDecimals,
    string memory _name,
    string memory _symbol
) ERC20(_name, _symbol)
```

The constructor initializes the aToken.

It:

```text
validates the addresses provider
validates the underlying asset
stores the addresses provider
retrieves LendingPoolCore
retrieves LendingPool
stores the underlying asset
stores the underlying asset decimals
initializes the ERC20 name and symbol
```

The addresses provider must be assigned before it is used:

```solidity
i_addressesProvider =
    LendingPoolAddressesProvider(
        _addressesProvider
    );

address coreAddress =
    i_addressesProvider.getLendingPoolCore();

address poolAddress =
    i_addressesProvider.getLendingPool();
```

The constructor also verifies that the returned core and pool addresses are not zero.

# Minting on Deposit

## `mintOnDeposit`

```solidity
function mintOnDeposit(
    address _account,
    uint256 _amount
) external onlyLendingPool
```

This function mints aTokens after a user deposits the underlying asset.

Only `LendingPool` can call it.

Before minting the new deposit, the function:

```text
materializes any interest already accrued
updates redirected-interest accounting
mints the new deposit amount
emits MintOnDeposit
```

## 1. Accumulate Existing Interest

```solidity
(
    ,
    ,
    uint256 balanceIncrease,
    uint256 index
) = _cumulateBalance(_account);
```

Before minting the new deposit, the function first converts the user's already accrued interest into stored aToken principal.

This prevents the new deposit from receiving interest for the period before it entered the protocol.

For example:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

The current balance is:

```text
current balance =
    100 × 1.05 / 1.00

current balance = 105 aDAI
```

Therefore:

```text
balance increase =
    105 - 100

balance increase = 5 aDAI
```

`_cumulateBalance()` mints the `5 aDAI` and updates the user's index to `1.05 ray`.

After cumulation:

```text
stored principal = 105 aDAI
user index = 1.05 ray
```

## 2. Update Redirected Balance

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    _account,
    balanceIncrease + _amount,
    0
);
```

If the user redirects interest, the recipient's redirected balance must increase by:

```text
interest just materialized
+
new deposit amount
```

The materialized interest is included because it is now part of the user's principal and will generate future interest.

The new deposit is included because it will also generate future interest.

If the user does not redirect interest, the internal function returns immediately.

## 3. Mint the Deposit Amount

```solidity
_mint(_account, _amount);
```

The user receives one aToken unit for each unit deposited.

Example:

```text
deposit 20 DAI
mint 20 aDAI
```

The new deposit is minted only after the old interest has been accumulated and the user index has been updated.

## 4. Emit `MintOnDeposit`

```solidity
emit MintOnDeposit(
    _account,
    _amount,
    balanceIncrease,
    index
);
```

The event records:

```text
the user receiving the aTokens
the deposited amount
the interest materialized before the deposit
the user's new index
```

# Example 1: Deposit Without Interest Redirection

Assume Alice has:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

Alice is not redirecting her interest.

She deposits another:

```text
20 DAI
```

`LendingPool` calls:

```solidity
aToken.mintOnDeposit(
    Alice,
    20 ether
);
```

## Step 1: Materialize Existing Interest

Before the new deposit is minted, `_cumulateBalance(Alice)` calculates Alice's current balance:

```text
current balance =
    100 × 1.05 / 1.00

current balance = 105 aDAI
```

The accrued interest is:

```text
balance increase =
    105 - 100

balance increase = 5 aDAI
```

`_cumulateBalance()` then:

```text
mints 5 aDAI to Alice
updates Alice's index to 1.05 ray
```

Alice's stored principal becomes:

```text
100 + 5 = 105 aDAI
```

## Step 2: Check Interest Redirection

`mintOnDeposit()` calls:

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    Alice,
    5 ether + 20 ether,
    0
);
```

Alice has no redirection address, so the internal function returns immediately.

No redirected balance is updated.

## Step 3: Mint the New Deposit

The function mints:

```text
20 aDAI
```

to Alice.

Alice's stored principal becomes:

```text
105 + 20 = 125 aDAI
```

## Step 4: Emit the Event

The event contains:

```text
_from = Alice
_value = 20 aDAI
_fromBalanceIncrease = 5 aDAI
_fromIndex = 1.05 ray
```

## Final State

```text
Alice stored principal = 125 aDAI
Alice user index = 1.05 ray
```

The sequence is:

```text
materialize 5 aDAI of existing interest
        ↓
update Alice's index to 1.05 ray
        ↓
mint 20 aDAI for the new deposit
```

The new `20 aDAI` does not receive interest for the period before Alice deposited it.

# Example 2: Deposit With Interest Redirection

Assume Alice has:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

Alice redirects the interest generated by her balance to Bob.

Bob's redirected balance is:

```text
100 aDAI
```

Alice deposits another:

```text
20 DAI
```

`LendingPool` calls:

```solidity
aToken.mintOnDeposit(
    Alice,
    20 ether
);
```

## Step 1: Materialize Alice's Existing Interest

Alice's current balance is:

```text
current balance =
    100 × 1.05 / 1.00

current balance = 105 aDAI
```

Therefore:

```text
balance increase =
    105 - 100

balance increase = 5 aDAI
```

`_cumulateBalance(Alice)`:

```text
mints 5 aDAI to Alice
updates Alice's index to 1.05 ray
```

Alice's stored principal becomes:

```text
100 + 5 = 105 aDAI
```

## Step 2: Update Bob's Redirected Balance

Because Alice redirects interest to Bob, the function calls:

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    Alice,
    5 ether + 20 ether,
    0
);
```

The redirected balance increases by:

```text
5 aDAI of materialized interest
+
20 aDAI from the new deposit

total added = 25 aDAI
```

Bob's redirected balance becomes:

```text
previous redirected balance = 100 aDAI
balance added = 25 aDAI

new redirected balance =
    100 + 25

new redirected balance = 125 aDAI
```

The `125 aDAI` still belongs to Alice.

It only represents the principal whose future interest is redirected to Bob.

## Step 3: Mint the New Deposit

The function mints:

```text
20 aDAI
```

to Alice.

Alice's stored principal becomes:

```text
105 + 20 = 125 aDAI
```

## Step 4: Emit the Event

The event contains:

```text
_from = Alice
_value = 20 aDAI
_fromBalanceIncrease = 5 aDAI
_fromIndex = 1.05 ray
```

## Final State

```text
Alice stored principal = 125 aDAI
Alice user index = 1.05 ray

Bob redirected balance = 125 aDAI
```

Alice owns the full `125 aDAI`.

Bob receives only the future interest generated by that balance.

The sequence is:

```text
materialize Alice's existing 5 aDAI interest
        ↓
update Alice's index to 1.05 ray
        ↓
add 5 + 20 aDAI to Bob's redirected balance
        ↓
mint 20 aDAI for Alice's new deposit
```


# Reading the User Balance

## `balanceOf`

The key to understanding `balanceOf()` is to separate three values:

- principal balance: tokens that belong to the user;
- redirected balance: principal owned by other users whose interest is assigned to this user;
- visible balance: the amount returned by `balanceOf()`.

A redirected balance is used only to calculate interest. Its principal must then be removed because it still belongs to the redirecting user.

```solidity
function balanceOf(
    address _user
) public view override returns (uint256)
```

This function returns the user's current interest-bearing balance.

It begins by reading:

```solidity
uint256 currentPrincipalBalance =
    super.balanceOf(_user);

uint256 redirectedBalance =
    s_redirectedBalances[_user];
```

These two values represent different things:

```text
currentPrincipalBalance =
    aToken principal owned by the user

redirectedBalance =
    principal owned by other users whose interest
    has been redirected to this user
```

`super.balanceOf()` returns only the stored ERC20 principal balance. It does not include interest accrued since the user's last index update.

The overridden `balanceOf()` calculates that interest dynamically.

For the following examples, assume:

```text
growth factor =
    current reserve normalized income
    ÷ user index
```

For example:

```text
user index = 1.00 ray
current normalized income = 1.10 ray

growth factor =
    1.10 / 1.00
    = 1.10
```

A growth factor of `1.10` means that balances used in the calculation have grown by 10% since the user's index was last updated.

## Empty Balance

```solidity
if (
    currentPrincipalBalance == 0
        && redirectedBalance == 0
) {
    return 0;
}
```

If the user has no principal balance and no redirected balance, there is nothing on which to calculate interest.

```text
principal balance = 0
redirected balance = 0

balanceOf = 0
```

This early return also avoids trying to calculate a cumulated balance for an account that has never had its user index initialized.

## Case 1: No Interest Redirection

Assume the user:

* owns a principal balance;
* does not redirect their interest;
* receives no redirected balance from anyone else.

The function executes:

```solidity
return _calculateCumulatedBalance(
    _user,
    currentPrincipalBalance
        + redirectedBalance
) - redirectedBalance;
```

Because the redirected balance is zero, this becomes:

```text
visible balance =
    cumulated principal balance
```

Assume:

```text
principal balance = 100
redirected balance = 0
growth factor = 1.10
```

The calculation is:

```text
cumulated balance =
    (100 + 0) × 1.10
    = 110

visible balance =
    110 - 0
    = 110
```

The returned balance contains:

```text
100 principal
 10 accrued interest
---
110 visible balance
```

This is the basic aToken case.

## Case 2: The User Receives Redirected Interest

```solidity
return _calculateCumulatedBalance(
    _user,
    currentPrincipalBalance
        + redirectedBalance
) - redirectedBalance;
```

Now assume Alice owns 100 aTokens and Bob redirects the interest generated by 50 of his aTokens to Alice.

For Alice:

```text
principal balance = 100
redirected balance = 50
growth factor = 1.10
```

The redirected balance does not mean that Alice owns Bob's 50 aTokens.

It means that Bob's 50 aTokens are temporarily included when calculating the interest that Alice should receive.

The function first calculates the growth of the combined amount:

```text
amount generating interest =
    Alice's principal + Bob's redirected principal

amount generating interest =
    100 + 50
    = 150
```

Apply the growth factor:

```text
cumulated balance =
    150 × 1.10
    = 165
```

The result contains:

```text
Alice's principal                    = 100
interest on Alice's principal        = 10
Bob's redirected principal           = 50
interest on Bob's redirected balance = 5
                                      ---
                                      165
```

Bob's 50 principal does not belong to Alice, so it must be subtracted:

```text
visible balance =
    165 - 50
    = 115
```

Alice's returned balance is therefore:

```text
Alice's principal                    = 100
interest on Alice's principal        = 10
interest redirected from Bob         = 5
                                      ---
visible balance                      = 115
```

The subtraction removes only Bob's principal. The 5 tokens of interest generated by Bob's principal remain assigned to Alice.


## User Redirects Their Own Interest

When the user has configured a nonzero redirection address, the function executes:

```solidity
return currentPrincipalBalance
    + _calculateCumulatedBalance(
        _user,
        redirectedBalance
    )
    - redirectedBalance;
```

The user's own principal is no longer passed to `_calculateCumulatedBalance()`.

Therefore, the user's principal does not generate interest for the user.

The formula is:

```text
visible balance =
    own principal
    + cumulated redirected balance
    - redirected principal
```

Equivalently:

```text
visible balance =
    own principal
    + interest generated by incoming redirected balances
```

## Case 3: The User Redirects Their Interest and Receives No Redirected Balance

Assume Bob owns 100 aTokens and redirects his interest to Alice.

For Bob:

```text
principal balance = 100
redirected balance = 0
growth factor = 1.10
redirection address = Alice
```

The calculation is:

```text
visible balance =
    100
    + (0 × 1.10)
    - 0

visible balance = 100
```

Bob's `balanceOf()` remains:

```text
100
```

His principal still belongs to him, but the 10 tokens of interest are assigned to Alice.

For Alice, assuming she has no principal of her own:

```text
principal balance = 0
redirected balance = 100
growth factor = 1.10
redirection address = address(0)
```

Alice's calculation is:

```text
cumulated balance =
    100 × 1.10
    = 110

visible balance =
    110 - 100
    = 10
```

The balances are therefore:

```text
Bob balanceOf   = 100
Alice balanceOf = 10
```

Together:

```text
100 principal + 10 interest = 110
```

The redirection changes who receives the interest. It does not transfer ownership of the principal.

## Case 4: Both Users Have Their Own Principal

Assume:

```text
Bob principal = 100
Alice principal = 50
```

Bob redirects his interest to Alice, and the growth factor is `1.10`.

### Bob

Bob redirects his interest and receives no incoming redirected balance:

```text
principal balance = 100
redirected balance = 0
```

His visible balance is:

```text
visible balance =
    100 + 0
    = 100
```

### Alice

Alice does not redirect her interest.

Her values are:

```text
principal balance = 50
redirected balance from Bob = 100
growth factor = 1.10
```

The calculation is:

```text
cumulated balance =
    (50 + 100) × 1.10
    = 165
```

Remove Bob's principal:

```text
visible balance =
    165 - 100
    = 65
```

Alice's balance consists of:

```text
Alice's principal               = 50
interest on Alice's principal   = 5
interest redirected from Bob    = 10
                                  --
visible balance                 = 65
```

The complete result is:

```text
Bob balanceOf   = 100
Alice balanceOf = 65
                  ---
total           = 165
```

This equals:

```text
total principal = 100 + 50 = 150
total interest  = 10 + 5  = 15
total balance   = 165
```

## Case 5: The User Redirects Their Own Interest but Also Receives Redirected Interest

Assume:

* Alice redirects her own interest to Carol;
* Bob redirects his interest to Alice;
* Alice owns 100 tokens;
* Bob has redirected a principal balance of 50 to Alice;
* the growth factor is `1.10`.

For Alice:

```text
principal balance = 100
redirected balance from Bob = 50
redirection address = Carol
```

Because Alice redirects her own interest, the second branch is used:

```text
visible balance =
    current principal
    + cumulated redirected balance
    - redirected balance
```

Calculate the growth of Bob's redirected balance:

```text
cumulated redirected balance =
    50 × 1.10
    = 55
```

Then:

```text
visible balance =
    100 + 55 - 50
    = 105
```

Alice's balance contains:

```text
Alice's principal             = 100
interest redirected from Bob  = 5
                                ---
visible balance               = 105
```

Alice does not receive the 10 tokens of interest generated by her own principal because that interest is redirected to Carol.

However, Alice still receives the 5 tokens of interest generated by Bob's redirected balance.

This is an important distinction:

```text
Alice redirects:
    interest generated by Alice's own principal

Alice does not redirect:
    interest generated by balances redirected to Alice
```

In Aave V1, interest generated only by an incoming redirected balance cannot itself be redirected again.


# Accumulating Interest

## `_cumulateBalance`

```solidity
function _cumulateBalance(
    address _user
)
    internal
    returns (
        uint256,
        uint256,
        uint256,
        uint256
    )
```

`_cumulateBalance` converts the interest already included in the user's dynamic `balanceOf()` into stored ERC20 principal.

Before this function is called, the user's balance may be divided into:

```text
stored principal
+
virtual accrued interest
```

After the function runs, the accrued interest is minted and becomes part of the stored principal.

The user's economic balance does not increase during this operation. The function only materializes interest that had already accrued.

It returns:

```text
previous principal balance
new principal balance
balance increase
new user index
```

### 1. Read the Stored Principal

```solidity
uint256 previousPrincipalBalance =
    super.balanceOf(_user);
```

`super.balanceOf()` reads the balance stored directly in the inherited ERC20 contract.

It does not include interest accrued since the user's last index update.

Example:

```text
stored principal = 100 aDAI
```

### 2. Calculate the Current Balance

The function calls the overridden `balanceOf()`:

```solidity
balanceOf(_user)
```

This returns the user's current interest-bearing balance by applying the reserve normalized income to the user's balance.

For a user without interest redirection:

```text
current balance =
    principal balance
    × current normalized income
    ÷ user index
```

Assume:

```text
previous principal balance = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

Then:

```text
current balance =
    100 × 1.05 / 1.00

current balance = 105 aDAI
```

### 3. Calculate the Balance Increase

```solidity
uint256 balanceIncrease =
    balanceOf(_user)
        - previousPrincipalBalance;
```

The balance increase is the interest accrued since the user's previous index checkpoint.

Using the previous example:

```text
current balance = 105 aDAI
previous principal balance = 100 aDAI

balance increase =
    105 - 100

balance increase = 5 aDAI
```

The `5 aDAI` already belongs to the user economically, but it has not yet been minted into ERC20 storage.

### 4. Mint the Accrued Interest

```solidity
_mint(_user, balanceIncrease);
```

The function mints the accrued interest and adds it to the stored principal.

Before minting:

```text
stored principal = 100 aDAI
dynamic balance = 105 aDAI
```

After minting:

```text
stored principal = 105 aDAI
dynamic balance = 105 aDAI
```

The user's current balance remains unchanged.

Only its representation changes:

```text
before:
    100 stored principal
    + 5 virtual interest

after:
    105 stored principal
    + 0 unapplied interest
```

Minting also increases the stored ERC20 supply by the same `balanceIncrease`.

### 5. Update the User Index

```solidity
uint256 index =
    s_userIndexes[_user] =
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        );
```

After materializing the interest, the user's index is moved to the current reserve normalized income.

Using the example:

```text
old user index = 1.00 ray
current normalized income = 1.05 ray
new user index = 1.05 ray
```

This records that all reserve growth up to `1.05 ray` has already been applied to the user.

Future interest is calculated only from this new checkpoint onward.

Immediately after the update:

```text
current balance =
    105 × 1.05 / 1.05

current balance = 105 aDAI
```

The division by the new user index prevents the previously accrued `5 aDAI` from being counted again.

### 6. Return the Updated Values

```solidity
return (
    previousPrincipalBalance,
    previousPrincipalBalance
        + balanceIncrease,
    balanceIncrease,
    index
);
```

Using the complete example, the returned values are:

```text
previousPrincipalBalance = 100 aDAI
newPrincipalBalance = 105 aDAI
balanceIncrease = 5 aDAI
index = 1.05 ray
```

## Complete Example

Assume Alice has:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

The function first calculates:

```text
current balance =
    100 × 1.05 / 1.00

current balance = 105 aDAI
```

Then:

```text
balance increase =
    105 - 100

balance increase = 5 aDAI
```

It mints the accrued interest:

```text
new stored principal =
    100 + 5

new stored principal = 105 aDAI
```

Finally, it updates the user index:

```text
new user index = 1.05 ray
```

The final state is:

```text
stored principal = 105 aDAI
current balance = 105 aDAI
user index = 1.05 ray
```

The user's value was already `105 aDAI` before the function ran. `_cumulateBalance()` only converted the virtual `5 aDAI` of accrued interest into stored principal.

## First Interaction

For a user with no balance:

```text
stored principal = 0
redirected balance = 0
user index = 0
```

`balanceOf()` returns zero before attempting to divide by the user index.

Therefore:

```text
previous principal balance = 0
current balance = 0
balance increase = 0
```

The function mints zero interest and sets the user's index to the current reserve normalized income.

This establishes the user's starting checkpoint before a new deposit is minted.


# Calculating the Cumulated Balance

## `_calculateCumulatedBalance`

```solidity
function _calculateCumulatedBalance(
    address _user,
    uint256 _balance
) internal view returns (uint256)
```

This function calculates a user's balance including the interest accrued since the user's index was last updated.

The formula is:

```text
cumulated balance =
    stored balance
    × current reserve normalized income
    ÷ user index
```

The implementation is:

```solidity
return _balance
    .wadToRay()
    .rayMul(
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        )
    )
    .rayDiv(s_userIndexes[_user])
    .rayToWad();
```

The ratio:

```text
current normalized income / user index
```

represents the reserve growth that occurred since the user's last balance update.

For example:

```text
user index = 1.02 ray
current normalized income = 1.071 ray

growth factor =
    1.071 / 1.02
    = 1.05
```

This means the reserve grew by 5% relative to the user's stored index.

For a stored balance of 100 tokens:

```text
cumulated balance =
    100 × 1.05
    = 105 tokens
```

### Example 1: the reserve has grown by 10%

```text
stored balance = 200 tokens
user index = 1.00 ray
current normalized income = 1.10 ray
```

First, calculate the growth factor:

```text
growth factor =
    1.10 / 1.00
    = 1.10
```

Then apply it to the stored balance:

```text
cumulated balance =
    200 × 1.10
    = 220 tokens
```

The user has earned:

```text
220 - 200 = 20 tokens
```

### Example 2: the reserve has grown, but the user entered later

```text
stored balance = 200 tokens
user index = 1.20 ray
current normalized income = 1.26 ray
```

The reserve normalized income has increased from `1.20` to `1.26`.

The relative growth is:

```text
growth factor =
    1.26 / 1.20
    = 1.05
```

Therefore:

```text
cumulated balance =
    200 × 1.05
    = 210 tokens
```

Even though the current normalized income is `1.26`, the user does not receive 26% interest.

The user only receives the 5% growth that occurred after their index was recorded at `1.20`.

### Example 3: two users with the same stored balance but different indexes

Assume both users have a stored balance of 100 tokens and the current normalized income is `1.20 ray`.

#### User A

```text
stored balance = 100 tokens
user index = 1.00 ray
current normalized income = 1.20 ray
```

```text
growth factor =
    1.20 / 1.00
    = 1.20
```

```text
cumulated balance =
    100 × 1.20
    = 120 tokens
```

User A has earned 20 tokens.

#### User B

```text
stored balance = 100 tokens
user index = 1.10 ray
current normalized income = 1.20 ray
```

```text
growth factor =
    1.20 / 1.10
    ≈ 1.090909
```

```text
cumulated balance =
    100 × 1.090909
    ≈ 109.0909 tokens
```

User B earns less because their index was updated later.

The current normalized income is the same for both users, but each user's balance depends on the normalized income stored in their own user index.

### Example 4: the user interacts multiple times

Assume the user initially has:

```text
stored balance = 100 tokens
user index = 1.00 ray
```

Later, the current normalized income becomes:

```text
current normalized income = 1.05 ray
```

Before updating the user's stored state, the balance is:

```text
cumulated balance =
    100 × 1.05 / 1.00
    = 105 tokens
```

When the user performs an action, the protocol may store:

```text
new stored balance = 105 tokens
new user index = 1.05 ray
```

Later, the normalized income increases again:

```text
current normalized income = 1.1025 ray
```

The new cumulated balance is:

```text
cumulated balance =
    105 × 1.1025 / 1.05
```

The relative growth factor is:

```text
1.1025 / 1.05 = 1.05
```

Therefore:

```text
cumulated balance =
    105 × 1.05
    = 110.25 tokens
```

The previously accrued interest becomes part of the stored balance, and the next calculation applies only the growth that happened after the latest index update.

### Example 5: normalized income contains cumulative reserve growth

Assume the reserve normalized income has grown as follows:

```text
reserve initialization: 1.00 ray
after first period:      1.10 ray
after second period:     1.21 ray
```

The value `1.21 ray` means the reserve has grown cumulatively by 21% since initialization.

However, assume the user index was recorded after the first period:

```text
stored balance = 100 tokens
user index = 1.10 ray
current normalized income = 1.21 ray
```

The user's growth factor is:

```text
growth factor =
    1.21 / 1.10
    = 1.10
```

Therefore:

```text
cumulated balance =
    100 × 1.10
    = 110 tokens
```

The user receives only the 10% growth from `1.10` to `1.21`, not the complete 21% growth since the reserve was initialized.

### Example 6: calculation using wad and ray units

Assume:

```text
stored balance = 100e18
user index = 1.02e27
current normalized income = 1.071e27
```

The stored balance starts in wad precision:

```text
100e18
```

It is converted to ray precision:

```text
100e18 × 1e9 = 100e27
```

The normalized income is then applied:

```text
100e27 × 1.071e27 / 1e27
= 107.1e27
```

The result is divided by the user index:

```text
107.1e27 × 1e27 / 1.02e27
= 105e27
```

Finally, the result is converted back to wad:

```text
105e27 / 1e9
= 105e18
```

Therefore, the returned balance is:

```text
105e18
```

which represents 105 tokens with 18 decimals.

The conversion from wad to ray is necessary because the reserve normalized income and the user index are stored using ray precision.

# Updating Redirected Balances

## `_updateRedirectedBalanceOfRedirectionAddress`

```solidity
function _updateRedirectedBalanceOfRedirectionAddress(
    address _user,
    uint256 _balanceToAdd,
    uint256 _balanceToRemove
) internal
```

This function updates the redirected balance of the address receiving `_user`'s interest.

It is used when the balance generating redirected interest changes because of actions such as:

```text
a new deposit
a redemption
a transfer
newly materialized interest
```

## 1. Read the Redirection Address

```solidity
address redirectionAddress =
    s_interestRedirectionAddresses[_user];
```

This reads the address currently receiving the interest generated by `_user`.

Example:

```text
Alice redirects interest to Bob

redirectionAddress = Bob
```

## 2. Return if There Is No Redirection

```solidity
if (redirectionAddress == address(0)) {
    return;
}
```

If `_user` is not redirecting interest, no redirected-balance accounting is required.

## 3. Cumulate the Recipient Balance

```solidity
(
    ,
    ,
    uint256 balanceIncrease,
    uint256 index
) = _cumulateBalance(redirectionAddress);
```

Before changing the redirected balance, the function materializes the interest already earned by the recipient.

This is necessary because the old redirected balance and the new redirected balance must not receive interest for the same period.

The recipient's current user index is also updated to the current reserve normalized income.

## 4. Update the Redirected Balance

```solidity
s_redirectedBalances[redirectionAddress] =
    s_redirectedBalances[redirectionAddress]
        + _balanceToAdd
        - _balanceToRemove;
```

The function adds or removes redirected principal.

It can:

```text
add a new deposit
add newly materialized interest
remove a redeemed amount
remove a transferred amount
```

## 5. Propagate Chained Redirection

```solidity
address targetOfRedirectionAddress =
    s_interestRedirectionAddresses[
        redirectionAddress
    ];

if (
    targetOfRedirectionAddress != address(0)
) {
    s_redirectedBalances[
        targetOfRedirectionAddress
    ] =
        s_redirectedBalances[
            targetOfRedirectionAddress
        ] + balanceIncrease;
}
```

The recipient may also redirect their own interest to another user.

In that case, the interest just materialized for the recipient becomes part of the balance generating interest for the next recipient.

Only `balanceIncrease` is propagated.

The original redirected principal is not propagated again because it still belongs to the first user.

## 6. Emit `RedirectedBalanceUpdated`

```solidity
emit RedirectedBalanceUpdated(
    redirectionAddress,
    balanceIncrease,
    index,
    _balanceToAdd,
    _balanceToRemove
);
```

The event records:

```text
the recipient whose redirected balance changed
the interest materialized for that recipient
the recipient's new index
the redirected balance added
the redirected balance removed
```

# Example 1: Adding a New Deposit to an Existing Redirection

Assume Alice redirects the interest generated by her aDAI balance to Bob.

The current state is:

```text
Alice principal balance = 100 aDAI
Alice redirection address = Bob

Bob redirected balance = 100 aDAI
Bob user index = 1.00 ray

current normalized income = 1.10 ray
```

The `100 aDAI` redirected balance does not belong to Bob.

It only means that Bob receives the interest generated by Alice's `100 aDAI`.

After the reserve normalized income grows from `1.00 ray` to `1.10 ray`, Bob has earned:

```text
cumulated redirected balance =
    100 × 1.10 / 1.00

cumulated redirected balance = 110 aDAI
```

Only the growth belongs to Bob:

```text
balance increase =
    110 - 100

balance increase = 10 aDAI
```

Alice then deposits another `50 DAI`.

`mintOnDeposit()` calls:

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    Alice,
    50 ether,
    0
);
```

## Step 1: Find Alice's Recipient

```text
redirectionAddress = Bob
```

## Step 2: Cumulate Bob's Existing Interest

```solidity
_cumulateBalance(Bob);
```

Bob's previously earned `10 aDAI` becomes stored principal.

Bob's user index becomes:

```text
1.10 ray
```

This ensures Alice's new `50 aDAI` does not receive interest for the period before it was deposited.

## Step 3: Add the New Redirected Balance

```text
previous redirected balance = 100 aDAI
balance added = 50 aDAI
balance removed = 0

new redirected balance =
    100 + 50 - 0

new redirected balance = 150 aDAI
```

## Final State

```text
Alice owns 150 aDAI principal

Bob owns:
    10 aDAI of materialized interest

Bob redirected balance:
    150 aDAI
```

From this point onward, Bob earns interest generated by Alice's full `150 aDAI`.

The sequence is:

```text
materialize interest earned on the old redirected balance
        ↓
update Bob's user index
        ↓
add Alice's new deposit
        ↓
future interest accrues on 150 aDAI
```

# Example 2: Chained Interest Redirection

Assume the following redirection chain:

```text
Alice redirects interest to Bob
Bob redirects his own interest to Charlie
```

The current state is:

```text
Alice principal balance = 100 aDAI
Alice redirection address = Bob

Bob redirected balance = 100 aDAI
Bob user index = 1.00 ray
Bob redirection address = Charlie

Charlie redirected balance = 0

current normalized income = 1.10 ray
```

Alice's `100 aDAI` generates interest for Bob.

Because Bob redirects his own interest to Charlie, the interest earned by Bob must ultimately generate interest for Charlie.

Alice then performs an action that calls:

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    Alice,
    0,
    0
);
```

## Step 1: Find Alice's Recipient

```text
redirectionAddress = Bob
```

## Step 2: Cumulate Bob's Balance

Bob has earned interest on Alice's redirected `100 aDAI`:

```text
cumulated redirected balance =
    100 × 1.10 / 1.00

cumulated redirected balance = 110 aDAI
```

Therefore:

```text
balanceIncrease =
    110 - 100

balanceIncrease = 10 aDAI
```

`_cumulateBalance(Bob)` mints the `10 aDAI` to Bob's stored principal and updates Bob's user index to `1.10 ray`.

## Step 3: Find Bob's Redirection Target

```solidity
address targetOfRedirectionAddress =
    s_interestRedirectionAddresses[Bob];
```

The result is:

```text
targetOfRedirectionAddress = Charlie
```

## Step 4: Propagate Bob's Materialized Interest

Because Bob redirects his own interest to Charlie:

```solidity
s_redirectedBalances[Charlie] =
    s_redirectedBalances[Charlie]
        + balanceIncrease;
```

The calculation is:

```text
Charlie's previous redirected balance = 0
Bob's balance increase = 10 aDAI

Charlie's new redirected balance =
    0 + 10

Charlie's new redirected balance = 10 aDAI
```

Only Bob's `10 aDAI` balance increase is propagated.

Alice's original `100 aDAI` is not added to Charlie's redirected balance because:

```text
Alice's principal generates interest for Bob
Bob's generated interest becomes principal owned by Bob
Bob redirects the interest generated by that new principal to Charlie
```

## Final State

```text
Alice owns:
    100 aDAI principal

Bob owns:
    10 aDAI materialized interest

Bob redirected balance:
    100 aDAI from Alice

Charlie redirected balance:
    10 aDAI from Bob
```

From this point onward:

```text
Alice's 100 aDAI continues generating interest for Bob

Bob's 10 aDAI materialized interest generates interest for Charlie
```

The chain is:

```text
Alice's principal
        ↓ generates interest
Bob receives 10 aDAI
        ↓ Bob redirects interest
Charlie's redirected balance increases by 10 aDAI
```

This mechanism propagates only newly materialized interest through the redirection chain, not the original principal.

# Reading the Total Supply

## `totalSupply`

```solidity
function totalSupply()
    public
    view
    override
    returns (uint256)
```

This function returns the current interest-bearing total supply.

It first reads the stored principal supply:

```solidity
uint256 currentSupplyPrincipal =
    super.totalSupply();
```

If no aTokens exist:

```solidity
if (currentSupplyPrincipal == 0) {
    return 0;
}
```

Otherwise, it applies the current reserve normalized income:

```solidity
return currentSupplyPrincipal
    .wadToRay()
    .rayMul(
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        )
    )
    .rayToWad();
```

Example:

```text
stored principal supply = 1,000 aDAI
normalized income = 1.05 ray

current total supply =
    1,000 × 1.05
    = 1,050 aDAI
```

Unlike an individual balance, `totalSupply()` does not divide by a user index because it represents the global deposit supply.

# First Deposit Example

Before the user's first deposit:

```text
stored principal = 0
redirected balance = 0
user index = 0
```

`balanceOf()` returns zero before performing any division.

`_cumulateBalance()` then sets the user index to the current reserve normalized income.

Finally, `mintOnDeposit()` mints the deposited amount.

Example:

```text
deposit = 100 DAI
current normalized income = 1.08 ray

stored principal = 100 aDAI
user index = 1.08 ray
```

The user does not receive interest for the period before the deposit.

# Later Deposit Example

Assume the user already has:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

The user deposits another `20 DAI`.

Before minting the new deposit:

```text
current balance =
    100 × 1.05 / 1.00
    = 105

balance increase =
    105 - 100
    = 5
```

`_cumulateBalance()` mints the `5 aDAI` interest and updates the user index to `1.05 ray`.

`mintOnDeposit()` then mints `20 aDAI`.

The final state is:

```text
stored principal = 125 aDAI
user index = 1.05 ray
```

# Virtual and Materialized Interest

Interest exists in two forms.

## Virtual Interest

Virtual interest is calculated dynamically by `balanceOf()`.

```text
virtual interest =
    current balance
    - stored principal
```

It is not yet stored in ERC20 state.

## Materialized Interest

Materialized interest is minted by `_cumulateBalance()`:

```solidity
_mint(_user, balanceIncrease);
```

Before cumulation:

```text
stored principal = 100
current balance = 105
```

After cumulation:

```text
stored principal = 105
current balance = 105
```

The user's economic value does not change. The function only converts already accrued virtual interest into stored principal.

# Summary

`AToken` represents a user's supplied position in Aave V1.

Its main accounting formula is:

```text
current balance =
    stored principal
    × current reserve normalized income
    ÷ user index
```

The main update sequence is:

```text
calculate accrued interest
        ↓
mint accrued interest
        ↓
update the user index
        ↓
perform the new deposit operation
```

This design allows balances to grow continuously while only updating storage when a state-changing operation occurs.
