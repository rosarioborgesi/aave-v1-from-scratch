# AToken

`AToken` is the interest-bearing ERC20 token used by Aave V1 to represent a user's deposit in a reserve.

When a user deposits an asset:

```text
user deposits the underlying asset
        ↓
LendingPoolCore holds the asset
        ↓
AToken mints an equivalent amount of aTokens
```

For example:

```text
deposit 100 DAI
receive 100 aDAI
```

The aToken balance increases over time as the reserve earns interest.

A useful mental model is:

```text
LendingPool
    = coordinates the deposit

LendingPoolCore
    = stores the reserve state and normalized income

AToken
    = represents the user's deposit and accrued interest
```

# Main Responsibilities

The current implementation of `AToken` has five main responsibilities:

```text
1. Mint aTokens when a user deposits.
2. Track each user's reserve-income checkpoint.
3. Calculate the user's current interest-bearing balance.
4. Materialize accrued interest during state-changing actions.
5. Track redirected interest balances.
```

# Protocol Relationships

## LendingPool

`LendingPool` is the only contract allowed to call privileged aToken functions such as:

```text
mintOnDeposit()
```

This prevents users from minting aTokens without depositing the underlying asset.

## LendingPoolCore

`LendingPoolCore` stores the reserve liquidity index and exposes:

```solidity
getReserveNormalizedIncome()
```

`AToken` uses this value to calculate how much the user's deposit has grown.

## Underlying Asset

Each aToken represents one underlying reserve asset.

Examples:

```text
aDAI represents DAI
aUSDC represents USDC
```

The underlying asset itself is held by `LendingPoolCore`, not by `AToken`.

# Access Control

## `onlyLendingPool`

```solidity
modifier onlyLendingPool() {
    if (msg.sender != address(i_pool)) {
        revert AToken__OnlyLendingPool();
    }
    _;
}
```

Only the configured `LendingPool` can call protected functions.

The current implementation uses this modifier on:

```text
mintOnDeposit()
```

# Stored State

## `i_underlyingAssetAddress`

```solidity
address private immutable i_underlyingAssetAddress;
```

Stores the underlying asset represented by the aToken.

For example:

```text
aDAI -> DAI address
```

The address is used when requesting the reserve normalized income from `LendingPoolCore`.

## `i_underlyingAssetDecimals`

```solidity
uint8 private immutable i_underlyingAssetDecimals;
```

Stores the decimals of the underlying asset.

The aToken overrides `decimals()` so that it uses the same decimal precision as the represented asset.

## `s_userIndexes`

```solidity
mapping(address user => uint256 lastNormalizedIncome)
    private s_userIndexes;
```

Stores the reserve normalized income last applied to each user.

The user index acts as a checkpoint.

For example:

```text
user index = 1.02 ray
current normalized income = 1.08 ray
```

Only the growth between `1.02` and `1.08` still needs to be applied to that user.

## `s_interestRedirectionAddresses`

```solidity
mapping(address => address)
    private s_interestRedirectionAddresses;
```

Stores the address receiving the interest generated by a user's balance.

A zero address means that the user is not redirecting interest.

## `s_redirectedBalances`

```solidity
mapping(address => uint256)
    private s_redirectedBalances;
```

Stores the amount of principal redirected to a user for interest calculation.

The recipient does not own this principal. They only receive the interest generated by it.

## `i_addressesProvider`

```solidity
LendingPoolAddressesProvider
    private immutable i_addressesProvider;
```

Stores the protocol addresses provider.

It is used during construction to retrieve `LendingPoolCore` and `LendingPool`.

## `i_core`

```solidity
LendingPoolCore private immutable i_core;
```

Stores the `LendingPoolCore` contract.

It is used to read the current reserve normalized income.

## `i_pool`

```solidity
LendingPool private immutable i_pool;
```

Stores the `LendingPool` contract.

It is used by the `onlyLendingPool` modifier.

# Constructor

```solidity
constructor(
    address _addressesProvider,
    address _underlyingAsset,
    uint8 _underlyingAssetDecimals,
    string memory _name,
    string memory _symbol
) ERC20(_name, _symbol)
```

The constructor initializes the aToken.

It:

```text
validates the addresses provider
validates the underlying asset
stores the addresses provider
retrieves LendingPoolCore
retrieves LendingPool
stores the underlying asset
stores the underlying asset decimals
initializes the ERC20 name and symbol
```

The addresses provider must be assigned before it is used:

```solidity
i_addressesProvider =
    LendingPoolAddressesProvider(
        _addressesProvider
    );

address coreAddress =
    i_addressesProvider.getLendingPoolCore();

address poolAddress =
    i_addressesProvider.getLendingPool();
```

The constructor also verifies that the returned core and pool addresses are not zero.

# Minting on Deposit

## `mintOnDeposit`

```solidity
function mintOnDeposit(
    address _account,
    uint256 _amount
) external onlyLendingPool
```

This function mints aTokens after a user deposits the underlying asset.

It performs four operations.

## 1. Accumulate Existing Interest

```solidity
(
    ,
    ,
    uint256 balanceIncrease,
    uint256 index
) = _cumulateBalance(_account);
```

Before minting the new deposit, the function first applies any interest already accrued by the user.

This prevents the new deposit from receiving interest for time before it entered the protocol.

## 2. Update Redirected Balance

```solidity
_updateRedirectedBalanceOfRedirectionAddress(
    _account,
    balanceIncrease + _amount,
    0
);
```

If the user redirects interest, the recipient's redirected balance increases by:

```text
accrued interest
+
new deposit amount
```

If no redirection exists, the internal function returns immediately.

## 3. Mint the Deposit Amount

```solidity
_mint(_account, _amount);
```

The user receives one aToken unit for each unit deposited.

Example:

```text
deposit 100 DAI
mint 100 aDAI
```

## 4. Emit `MintOnDeposit`

```solidity
emit MintOnDeposit(
    _account,
    _amount,
    balanceIncrease,
    index
);
```

The event records:

```text
the deposited amount
the accrued interest materialized before the deposit
the new user index
```

# Reading the User Balance

## `balanceOf`

```solidity
function balanceOf(
    address _user
) public view override returns (uint256)
```

This function returns the user's current interest-bearing balance.

It begins by reading:

```solidity
uint256 currentPrincipalBalance =
    super.balanceOf(_user);

uint256 redirectedBalance =
    s_redirectedBalances[_user];
```

`super.balanceOf()` returns only the stored ERC20 principal balance.

The overridden `balanceOf()` then adds accrued interest.

# Empty Balance

```solidity
if (
    currentPrincipalBalance == 0
        && redirectedBalance == 0
) {
    return 0;
}
```

If the user has no principal and no redirected balance, the function returns zero.

This also prevents division by a zero user index.

# User Does Not Redirect Interest

```solidity
if (
    s_interestRedirectionAddresses[_user]
        == address(0)
) {
    return _calculateCumulatedBalance(
        _user,
        currentPrincipalBalance
            + redirectedBalance
    ) - redirectedBalance;
}
```

If the user does not redirect interest, both their own principal and any incoming redirected balance generate interest for them.

The redirected principal is subtracted because it does not belong to the user.

Example:

```text
user principal = 100
redirected balance = 50
growth factor = 1.10

cumulated balance =
    (100 + 50) × 1.10
    = 165

visible balance =
    165 - 50
    = 115
```

The user receives:

```text
10 interest on their own principal
5 interest from redirected principal
```

# User Redirects Their Own Interest

```solidity
return currentPrincipalBalance
    + _calculateCumulatedBalance(
        _user,
        redirectedBalance
    )
    - redirectedBalance;
```

If the user redirects their own interest, their principal does not generate interest for them.

They can still receive interest from balances redirected to them by other users.

# Accumulating Interest

## `_cumulateBalance`

```solidity
function _cumulateBalance(
    address _user
)
    internal
    returns (
        uint256,
        uint256,
        uint256,
        uint256
    )
```

This function converts accrued interest into stored ERC20 principal.

It returns:

```text
previous principal balance
new principal balance
balance increase
new user index
```

## 1. Read the Stored Principal

```solidity
uint256 previousPrincipalBalance =
    super.balanceOf(_user);
```

This is the balance already stored in ERC20 state.

## 2. Calculate the Balance Increase

```solidity
uint256 balanceIncrease =
    balanceOf(_user)
        - previousPrincipalBalance;
```

Example:

```text
stored principal = 100
current balance = 105

balance increase = 5
```

## 3. Mint the Accrued Interest

```solidity
_mint(_user, balanceIncrease);
```

The accrued interest becomes stored principal.

After minting:

```text
old stored principal = 100
balance increase = 5
new stored principal = 105
```

## 4. Update the User Index

```solidity
uint256 index =
    s_userIndexes[_user] =
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        );
```

The user's checkpoint is moved to the current reserve normalized income.

Future interest is calculated only from this new index onward.

# Calculating the Cumulated Balance

## `_calculateCumulatedBalance`

```solidity
function _calculateCumulatedBalance(
    address _user,
    uint256 _balance
) internal view returns (uint256)
```

This function calculates a balance including the interest accrued since the user's last index update.

The formula is:

```text
cumulated balance =
    balance
    × current reserve normalized income
    ÷ user index
```

The implementation is:

```solidity
return _balance
    .wadToRay()
    .rayMul(
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        )
    )
    .rayDiv(s_userIndexes[_user])
    .rayToWad();
```

Example:

```text
balance = 100
user index = 1.02 ray
current normalized income = 1.071 ray

cumulated balance =
    100 × 1.071 / 1.02
    = 105
```

The balance is converted from wad to ray because the reserve indexes use ray precision.

# Updating Redirected Balances

## `_updateRedirectedBalanceOfRedirectionAddress`

```solidity
function _updateRedirectedBalanceOfRedirectionAddress(
    address _user,
    uint256 _balanceToAdd,
    uint256 _balanceToRemove
) internal
```

This function updates the redirected balance of the address receiving `_user`'s interest.

## 1. Read the Redirection Address

```solidity
address redirectionAddress =
    s_interestRedirectionAddresses[_user];
```

## 2. Return if There Is No Redirection

```solidity
if (redirectionAddress == address(0)) {
    return;
}
```

## 3. Cumulate the Recipient Balance

```solidity
(
    ,
    ,
    uint256 balanceIncrease,
    uint256 index
) = _cumulateBalance(redirectionAddress);
```

The recipient's already accrued interest is materialized before changing the redirected principal.

This prevents a newly added redirected balance from receiving interest for time before it was added.

## 4. Update the Redirected Balance

```solidity
s_redirectedBalances[redirectionAddress] =
    s_redirectedBalances[redirectionAddress]
        + _balanceToAdd
        - _balanceToRemove;
```

The function can:

```text
add a new deposit
add newly materialized interest
remove a redeemed amount
remove a transferred amount
```

## 5. Propagate Chained Redirection

If the recipient also redirects interest, their newly materialized balance increase is added to the next recipient's redirected balance.

## 6. Emit `RedirectedBalanceUpdated`

```solidity
emit RedirectedBalanceUpdated(
    redirectionAddress,
    balanceIncrease,
    index,
    _balanceToAdd,
    _balanceToRemove
);
```

# Token Decimals

## `decimals`

```solidity
function decimals()
    public
    view
    override
    returns (uint8)
```

Returns the decimals of the underlying asset.

This preserves the same unit precision between the reserve asset and its aToken.

# Reading the Total Supply

## `totalSupply`

```solidity
function totalSupply()
    public
    view
    override
    returns (uint256)
```

This function returns the current interest-bearing total supply.

It first reads the stored principal supply:

```solidity
uint256 currentSupplyPrincipal =
    super.totalSupply();
```

If no aTokens exist:

```solidity
if (currentSupplyPrincipal == 0) {
    return 0;
}
```

Otherwise, it applies the current reserve normalized income:

```solidity
return currentSupplyPrincipal
    .wadToRay()
    .rayMul(
        i_core.getReserveNormalizedIncome(
            i_underlyingAssetAddress
        )
    )
    .rayToWad();
```

Example:

```text
stored principal supply = 1,000 aDAI
normalized income = 1.05 ray

current total supply =
    1,000 × 1.05
    = 1,050 aDAI
```

Unlike an individual balance, `totalSupply()` does not divide by a user index because it represents the global deposit supply.


# First Deposit Example

Before the user's first deposit:

```text
stored principal = 0
redirected balance = 0
user index = 0
```

`balanceOf()` returns zero before performing any division.

`_cumulateBalance()` then sets the user index to the current reserve normalized income.

Finally, `mintOnDeposit()` mints the deposited amount.

Example:

```text
deposit = 100 DAI
current normalized income = 1.08 ray

stored principal = 100 aDAI
user index = 1.08 ray
```

The user does not receive interest for the period before the deposit.

# Later Deposit Example

Assume the user already has:

```text
stored principal = 100 aDAI
user index = 1.00 ray
current normalized income = 1.05 ray
```

The user deposits another `20 DAI`.

Before minting the new deposit:

```text
current balance =
    100 × 1.05 / 1.00
    = 105

balance increase =
    105 - 100
    = 5
```

`_cumulateBalance()` mints the `5 aDAI` interest and updates the user index to `1.05 ray`.

`mintOnDeposit()` then mints `20 aDAI`.

The final state is:

```text
stored principal = 125 aDAI
user index = 1.05 ray
```

# Virtual and Materialized Interest

Interest exists in two forms.

## Virtual Interest

Virtual interest is calculated dynamically by `balanceOf()`.

```text
virtual interest =
    current balance
    - stored principal
```

It is not yet stored in ERC20 state.

## Materialized Interest

Materialized interest is minted by `_cumulateBalance()`:

```solidity
_mint(_user, balanceIncrease);
```

Before cumulation:

```text
stored principal = 100
current balance = 105
```

After cumulation:

```text
stored principal = 105
current balance = 105
```

The user's economic value does not change. The function only converts already accrued virtual interest into stored principal.

# Summary

`AToken` represents a user's supplied position in Aave V1.

Its main accounting formula is:

```text
current balance =
    stored principal
    × current reserve normalized income
    ÷ user index
```

The main update sequence is:

```text
calculate accrued interest
        ↓
mint accrued interest
        ↓
update the user index
        ↓
perform the new deposit operation
```

This design allows balances to grow continuously while only updating storage when a state-changing operation occurs.
